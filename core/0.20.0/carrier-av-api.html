Arctos Carrier AV APIs¶

Data Types¶

IOEXAvRateControlMode¶

enum IOEXAvRateControlMode¶

IOEXAvRateControlMode defines the rate control types used in toxav codec.

Values:

enumerator IOEXAvRateControlMode_VBR¶

In VBR mode, bitrate is variable.

enumerator IOEXAvRateControlMode_CBR¶

In CBR mode, encoder will try to keep a fixed bitrate (within a constraint range).

Although it won’t force all frames to be exactly the same size.

IOEXAvOptions¶

struct IOEXAvOptions¶

IOEXAvOptions defines some codec options for tuning.

Public Members

unsigned int encoder_threads¶

Number of threads used for multi-threaded encoding.

Suggest to be number of real CPU cores - 1.

unsigned int decoder_threads¶

Number of threads used for multi-threaded decoding.

Suggest to be number of real CPU cores - 1.

unsigned int encode_deadline¶

Max time to encode a frame in us.

Usually, longer deadline results in better quality.

Best quality: set to 0 (no deadline).

Good quality: set to (1000000 / max target fps).

Real time quality: set to 1.

unsigned int decode_deadline¶

Max time to decode a frame in us.

Usually, longer deadline results in better quality.

Best quality: set to 0 (no deadline).

Good quality: set to (1000000 / max target fps).

Real time quality: set to 1.

int cpu_used_value¶

The meaning of this value depends on encode/decode deadline value.

When using Best Quality mode (deadline = 0), this value has no meaning.

When using Good Quality mode (between 1000000 to 2), this value can be used to set further speed steps. Range from 0 ~ 5. The lower the number, the better quality.

When using Real Time mode (deadline = 1), this value controls max CPU utilisation: target CPU utilisation = (100 * ( 16 - cpu_used_value ) / 16)%.

So for example, cpu_used_value = 4 means target CPU utilisation = 75%.

IOEXAvRateControlMode rate_control_mode¶

Rate control mode.

See IOEXAvRateControlMode.

unsigned int max_keyframe_interval¶

Max interval between 2 keyframes.

The value is in frames. For example, say FPS = 20 and max_keyframe_interval = 40, then we will have a keyframes every 2 seconds. Note that this value is not mandatory. Codec could insert a keyframe within this interval automatically.

Set this value to 0 implies every frames are keyframes.

unsigned int rate_control_target_bitrate¶

Target encode bitrate in kbps for rate control.

This value here is an initial value. You can overwrite this value when using other IOEX AV APIs like IOEX_av_set_video_bitrate().

unsigned int allow_lagged_encode_frames¶

Determine how many future frames encoder can keep in order to encode current frame.

This will introduce lags when encode, but could improve frame quality.

Possible range 0~25.

unsigned int dropframe¶

Specify a buffer fullness threshold at which the encoder starts to drop frames in percentage.

Set 0 to disable drop frame feature.

bool allow_resize¶

Specify if encoder should resize the frame if the internal buffer meets certain constraints.

See resize_up_threshhold and resize_down_threshold.

unsigned int resize_up_threshold¶

This threshold is described as a percentage of the target data buffer.

When the data buffer rises above this percentage of fullness (percentage of buffer_optimal_size), the encoder will step up to a higher resolution version of the frame.

unsigned int resize_down_threshold¶

This threshold is described as a percentage of the target data buffer.

When the data buffer fells below this percentage of fullness (percentage of buffer_optimal_size), the encoder will step down to a lower resolution version of the frame.

unsigned int rate_control_undershoot_bits¶

Expressed as a percentage of the target bitrate, controls the maximum allowed adaptation speed of the codec.

This factor controls the maximum amount of bits that can be subtracted from the target bitrate in order to compensate for prior overshoot.

unsigned int rate_control_overshoot_bits¶

Expressed as a percentage of the target bitrate, controls the maximum allowed adaptation speed of the codec.

This factor controls the maximum amount of bits that can be added to the target bitrate in order to compensate for prior undershoot.

unsigned int buffer_size¶

Max decoder buffer size which indicates the amount of data that can be buffered.

The unit of this value is ms. For example, buffer_size = 5000 means that decoder will buffer 5000ms worth data before decoding.

Note that buffer_size >= buffer_optimal_size >= buffer_initial_size.

unsigned int buffer_initial_size¶

Initial decoder buffer size which indicates the amount of data that can be buffered, prior to playback.

The unit of this value is ms.

unsigned int buffer_optimal_size¶

Optimal decoder buffer size which indicates the amount of data that decoder should try to maintain.

The unit of this value is ms.

unsigned int initial_keyframe_count¶

Number of keyframes encoder should send when initial a session.

By sending more keyframes at the beginning, we could have a better, stable image at start.

IOEXAvCallStatus¶

enum IOEXAvCallStatus¶

Status of an AV call.

Values:

enumerator IOEXAvCallStatus_None¶

The call is not setup or status unknown.

enumerator IOEXAvCallStatus_Pending¶

The call is waiting for user’s approval.

enumerator IOEXAvCallStatus_Waiting¶

The call is waiting for response from the other peer.

enumerator IOEXAvCallStatus_Running¶

The call is running.

enumerator IOEXAvCallStatus_Paused¶

The call is paused.

enumerator IOEXAvCallStatus_Finished¶

The call is finished or canceled.

IOEX_av_call¶

int IOEX_av_call(IOEXCarrier *carrier, const char *friend_id, uint32_t audio_bitrate, uint32_t video_bitrate)¶

An application-defined function that send a av request to friend.

Parameters

  • carrier – [in] A handle to the Carrier node instance.

  • friend_id – [in] ID of the friend.

  • audio_bitrate – [in] Our audio transmitting bitrate in kbps. Set 0 to disable audio.

  • video_bitrate – [in] Our video transmitting bitrate in kbps. Set 0 to disable video.

Returns

0 if the request successfully sent. Otherwise, return -1, and a specific error code can be retrieved by calling IOEX_get_error().

IOEX_av_send_audio¶

int IOEX_av_send_audio(IOEXCarrier *carrier, const char *friend_id, const int16_t *pcm, size_t sample_count, uint8_t channels, uint32_t sampling_rate)¶

An application-defined function that sends an audio frame to a friend.

The expected format of the PCM data is: [s1c1][s1c2][…][s2c1][s2c2][…]… Meaning: sample 1 for channel 1, sample 1 for channel 2, … For mono audio, this has no meaning, every sample is subsequent. For stereo, this means the expected format is LRLRLR… with samples for left and right alternating.

Parameters

  • carrier – [in] A handle to the Carrier node instance.

  • friend_id – [in] ID of the friend.

  • pcm – [in] An array of audio samples. The size of this array must be sample_count * channels.

  • sample_count – [in] Number of samples in this frame. Valid numbers here are ((sample rate) * (audio length) / 1000), where audio length can be 2.5, 5, 10, 20, 40 or 60 millseconds.

  • channels – [in] Number of audio channels. Supported values are 1 and 2.

  • sampling_rate – [in] Audio sampling rate used in this frame. Valid sampling rates are 8000, 12000, 16000, 24000, or 48000.

Returns

0 if the request successfully sent. Otherwise, return -1, and a specific error code can be retrieved by calling IOEX_get_error().
